## P0 — Ship blockers

### B3 — Banner renders ~600 individual Ink Text elements [FIXED]

`src/components/Banner.tsx` renders each character as a separate `<Text color={...}>` node.
Full REEVES AGENTS art is 12 lines × ~50 chars = ~600 Ink elements. Every state change in
the parent screen triggers a full reconcile of all 600 nodes: visible flicker, slow
initial paint, potential visual artifacts on narrow terminals.

Fix: compute gradient-colored lines using chalk ANSI codes, then render each line as a
single `<Text>` node. Reduces Ink element count from ~600 to ~12.

Files: `src/components/Banner.tsx`, `src/brand/banner.ts`

---

### B1 — Shell injection in spawn.ts [FIXED]

`spawn.ts` builds tmux commands via template strings passed to `execSync`:

```typescript
execSync(`tmux send-keys -t ${TMUX_SESSION}:${name} '${cmdStr}' Enter`)
```

Single quotes in `name`, `cmdStr`, or `start_prompt` break the shell command silently.
A prompt like "don't forget auth" causes send-keys to fail with no error surfaced to the user.

Fix: replace all tmux `execSync` calls with `execFileSync('tmux', [...args])`. The args
array is passed directly to tmux without shell interpretation. For `start_prompt`, use
`tmux load-buffer -` (stdin) + `paste-buffer` to avoid quoting entirely.

Files: `src/launcher/spawn.ts`

---

### B2 — start_prompt sent via fire-and-forget setTimeout [FIXED]

```typescript
if (req.start_prompt) {
  setTimeout(() => {
    execSync(`tmux send-keys ... '${req.start_prompt}' Enter`)
  }, 1000)
}
```

`spawn()` returns the Session synchronously. The setTimeout fires 1 second later with no
guarantee the provider CLI is ready to accept input. If it is not ready, the prompt is
sent to the shell prompt, not to the AI session, and there is no retry or error.

Fix: switch to `execFileSync('tmux', ['load-buffer', '-'], { input: prompt })` +
`paste-buffer` inside the same setTimeout. At least eliminates the injection. Delay
increased to 1500ms. Add a `REEVES_INITIAL_PROMPT_DELAY_MS` env override for slower machines.

Files: `src/launcher/spawn.ts`

## P1 — Core gaps

### B6 — Store never updated on spawn or orchestrate [FIXED]

`setLastSpawn`, `setLastOrchestrate`, `addRecentSession` in `src/state/store.ts` all exist
and are tested. Nothing calls them. The entire history/state layer is dead:
- `store.last_spawn` stays at defaults across restarts
- `store.recent_sessions` is always empty
- `store.history.spawned_total` never increments
- Presets are saved/loaded but the spawn form never pre-populates from `last_spawn`

Fix: call `setLastSpawn(...)` and `addRecentSession(id)` at the end of `spawn()`.
Call `setLastOrchestrate(...)` at the end of `orchestrate()`.

Files: `src/launcher/spawn.ts`, `src/launcher/orchestrate.ts`

---

### F5 — Spawn and Orchestrate forms reset to defaults on every launch

Spawn form always initializes from `loadConfig()` provider defaults. It ignores
`store.last_spawn` entirely. Users who spawn the same provider/model/tag combination
repeatedly have to re-enter it every time.

Fix: on mount, load `store.last_spawn` and pre-populate form fields. User can override
any field as before.

Files: `src/screens/Spawn.tsx`, `src/screens/Orchestrate.tsx`

---

### F9 / F2 — No way to attach to a running session from the TUI

Sessions screen shows sessions and supports peek and kill, but there is no attach action.
Users must manually type `tmux attach -t reevesagents:<window>` in another terminal.

Fix: add `a` key binding in Sessions screen that prints the attach command to the status
bar / hint area and offers to switch to the tmux window if possible. On macOS, can use
`open-x11` or just display the command clearly.

Files: `src/screens/Sessions.tsx`

---

### F6 — Sessions screen does not auto-refresh

Sessions are loaded once on mount. New sessions spawned in the same TUI instance do not
appear until the user presses `r`. The screen has no timer.

Fix: add a `useEffect` with `setInterval(refresh, 5000)` and cleanup. 5-second refresh
interval is fast enough to feel live, slow enough to not hammer the filesystem.

Files: `src/screens/Sessions.tsx`

---

### B4 — CommandBar.tsx is dead code with duplicate SLASH_ROUTES [FIXED]

`src/components/CommandBar.tsx` defines its own `SLASH_ROUTES` constant that duplicates
the one in `useScreenNav.ts`. No screen imports or renders `CommandBar`. All screens inline
the command bar using `useScreenNav`'s `cmdMode`/`cmdValue`/`cmdError` state.

This file is unreachable code that will silently drift out of sync with `useScreenNav`.

Fix: delete `CommandBar.tsx`.

Files: `src/components/CommandBar.tsx` (delete)

---

### B15 — Doctor checks Node version via subprocess

`checkNodeVersion()` in `doctor.ts` runs `execSync('node -v')` to get the version. We are
already running in Node; `process.version` gives the same information with no subprocess.

Fix: replace `execSync('node -v')` with `process.version.slice(1)` and remove the
`execSync` import if it is no longer needed.

Files: `src/launcher/doctor.ts`

---

### F10 — Settings screen exposes only auth and key_env

Settings lets users set auth mode and key env var per provider. Missing:
- `default_model` per provider
- `default_permissions` per provider
- `default_effort` per provider
- `base_url` (shown/editable only when auth=custom)

The config schema has all these fields. The Settings form does not expose them.

Files: `src/screens/Settings.tsx`

---

### B8 — No tmux window name conflict detection

`spawn.ts` calls `tmux new-window -n <name>` without checking if a window with that name
already exists. If it does, `new-window` fails silently (the `stdio: 'ignore'` swallows
the error), but the session is still written to the registry pointing at a non-existent
window.

Fix: before creating the window, run `tmux list-windows -t reevesagents` and check for
conflicts. If the name conflicts, append `-2`, `-3`, etc. until a free name is found.

Files: `src/launcher/spawn.ts`

## P2 — Missing screens and commands

### F1 — No /top live monitor screen

The original spec identifies `/top` as the only auto-refreshing screen. Users need a way
to see all running sessions, their ages, providers, and last-seen timestamps in one view,
without entering the Sessions screen or manually running `tmux list-windows`.

Required behavior: auto-refreshes every 5 seconds. Shows session ID, provider badge,
name/tag, age, last_seen_at delta, and status (active vs stale). Press `a` on a row to
attach, `k` to kill.

Files: `src/screens/Top.tsx` (new), `src/hooks/useScreenNav.ts` (add `/top` route),
`src/router.tsx` (register screen), `src/state/types.ts` (add 'Top' to ScreenName)

---

### F12 — No reevesagents attach CLI subcommand

From the command line there is no way to attach to a session by ID. Users must look up
the tmux window name manually.

Fix: add `reevesagents attach <id>` that runs
`tmux switch-client -t reevesagents:<window>` or prints the attach command if not in a
tmux context.

Files: `src/cli.ts`

---

### F4 — No preset management UI

`addPreset`, `removePreset`, `loadState` are implemented and tested in store.ts. The TUI
has no way to save a spawn/orchestrate config as a preset or run a saved preset.

Minimum: in the Spawn screen, add a `[save as preset]` button after successful spawn.
In the Home screen, show saved presets with a key to run them.

Files: `src/screens/Spawn.tsx`, `src/screens/Home.tsx`, `src/state/store.ts`

---

### F11 — CLI spawn output does not include attach instructions

`reevesagents spawn ...` prints a JSON blob. Human-readable attach instructions are not
included. Users have to parse the JSON to find `tmux_session` and `tmux_window`.

Fix: after the JSON block, print `# attach: tmux attach -t <session>:<window>` as a
comment line.

Files: `src/cli.ts`

## P3 — Polish and nice-to-haves

### F7 / G7 — No goodbye message on exit

PROJECT_BIBLE specified a 30-language goodbye message rotating on exit. Currently the app
exits silently. Trivial to add: intercept `process.on('exit')` or call a farewell
function before `exit()` in `useScreenNav`.

---

### G15 — No error boundary

If any screen throws an unhandled React error, Ink crashes with a raw stack trace.
An `<ErrorBoundary>` in `Router` would catch this and show a clean error with a "press r
to restart" prompt.

---

### G9 — Orchestrate screen: post-spawn summary needs attach commands

After orchestrating, the success view shows session IDs and names but not the tmux attach
command for each. Users cannot quickly attach to check progress.

---

### G5 — Home screen shows session count but not last session detail

Show the most recently spawned session's ID, provider, and name on the Home screen, not
just a total count.

---

### G10 — Sessions screen: provider shown as plain text

Provider column should use a colored badge: cc=blue, codex=green, gemini=yellow.

---

### G13 — No progress indicator during spawn

`spawn()` calls several tmux commands that take ~200ms total. A brief inline "spawning..."
state between pressing `[spawn]` and seeing the result would improve perceived performance.

---

### G18 — Spawn form: no validation on name field

User-entered names go directly into the tmux window name. Spaces and tmux-special
characters (`:`, `.`) cause `new-window` to fail silently. Add a validation pass before
submitting.

---

### G17 — Terminal theme: hardcoded dark-terminal palette

The blue gradient (`#3a7ad8` to `#a5cdf7`) is calibrated for dark terminals. On a light
terminal it renders as low-contrast. A `NO_COLOR` env check and a fallback bold-text-only
mode would fix this.

---

### G14 — Doctor: tmux version check does not parse version number

The tmux check currently shows `tmux 3.3a` as the detail string. It should parse the
version and warn if below 3.0 (which does not support `display-popup` and other commands
we may use in the future).
